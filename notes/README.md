# Notes

Generated reports — daily, weekly, monthly, yearly summaries the system produces during check-ins. Treat this folder as output, not source: the canonical state lives in `5-actions/_data/`, `6-habits/_data/`, `data/`, etc. These files are rendered views.

## Subfolders

| Folder | Generated by | Cadence |
|---|---|---|
| `daily/` | `pfc-morning-checkin` and `pfc-evening-checkin` | One file per day (`YYYY-MM-DD.md`) |
| `weekly/` | `pfc-weekly-checkin` | One file per week (Sunday-driven) |
| `monthly/` | `pfc-monthly-checkin` | One file per month |
| `yearly/` | `pfc-yearly-checkin` | One file per year |

## What's in a note

The shape varies by cadence, but the common pattern:

- A header with the date range
- Summary of what happened (tasks completed, habits logged, day ratings, project velocity)
- Reflection prompts the user answered during the check-in
- Open questions and next-step intentions

The full content of each cadence is documented in the matching skill file (`.agents/skills/pfc-{morning,evening,weekly,monthly,yearly}-checkin/SKILL.md`).

## Reading vs. editing

Notes are mostly read-only after generation — they're a record of what was true at the time of the check-in. You can edit them to add later reflections, but the system itself won't re-render them.

If you find yourself wanting to *change* a generated note rather than add to it, the underlying data probably needs to change instead (e.g. fix the task status in `tasks.ndjson` rather than the summary in the daily note).

## Why notes are markdown, not NDJSON

NDJSON is for *records the system queries*. Notes are *narrative summaries the human reads*. Different surface, different format. Markdown is the right tool for the narrative side — easy to skim, easy to grep, easy to drop into the system's second-brain folders (`2-areas/<area>/`) if a note has lasting relevance.
